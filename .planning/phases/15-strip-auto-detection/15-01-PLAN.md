<context>
## Current State

The plugin currently auto-detects assets by walking the layout tree (`identifyAssets` in identify.ts) and detecting vector-only groups (`detectCompositions` in detect-composition.ts). These feed into `exportAssets` in export.ts, which calls Figma API to render and download them. The brief generator in generate.ts has composition-specific logic (compositionNodeIds, [Illustration] collapsed subtrees, assetTypeLabel for 'composition'/'component'). MainView.tsx passes rootNodes, imageFills, and instancesWithText to the export pipeline.

## Target State

After this plan:
- `identify.ts`, `identify.test.ts`, `detect-composition.ts`, `detect-composition.test.ts` are deleted
- `export.ts` is hollowed out to preview-only (no asset identification or batch download)
- `types.ts` assetType simplified to `'icon' | 'image'` (no 'composition' | 'component')
- `download.ts` inline assetType unions narrowed to `'icon' | 'image'`
- `generate.ts` stripped of all composition logic
- `extract.ts` stripped of `collectImageFillsFromRawTree` and `collectInstancesWithText`
- `MainView.tsx` simplified caller to match new export.ts signature
- All tests pass

- **Hollow out export.ts**: Remove identifyAssets/detectCompositions calls but keep preview PNG generation and download pipeline. Phase 17 will refactor to accept ManualAsset[].
- **Keep breadcrumb.ts**: May be useful for Phase 18. Delete later if not needed.

- **Delete identify.ts** (306 lines) and its test file entirely.
- **Delete detect-composition.ts** (208 lines) and its test file entirely.

- **Keep AssetEntry type** for now (it is imported by sanitize.ts; Phase 16 will replace with ManualAsset).
- **Leave `imageFills` in DesignTokens/collectTokens** (token-level metadata, not asset pipeline). Only remove `collectImageFillsFromRawTree` from extract.ts.
- **Tests**: All must pass. Delete test files alongside modules. Fix generate.test.ts references.
